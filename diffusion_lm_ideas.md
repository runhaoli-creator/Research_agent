## Idea 1: Mask-Adaptive Attention — An Architecture Designed for Iterative Denoising

**Core insight:** Every diffusion LM uses standard bidirectional Transformer attention regardless of the masking state. But attending to MASK tokens is wasted compute — they carry zero information. At early denoising steps (90% masked), 90% of attention is on uninformative tokens. We propose mask-adaptive attention where the attention pattern dynamically adjusts based on which tokens are revealed vs. masked: revealed tokens attend fully to each other, masked tokens attend only to revealed tokens (not to other masked tokens), and attention weights are re-normalized over the informative subset.

**Novelty verification (double-checked April 2026):** "Revealing the Attention Floating Mechanism" (arXiv 2601.07894, Jan 2026) ANALYZES how attention floats in MDMs (weaker, less concentrated patterns) but proposes NO architectural intervention. Soft-Masked DLM (2510.17206) enriches MASK embeddings but doesn't modify attention mechanics. AdaBlock-dLLM (2509.26432) does semantic-aware block-size decoding, not attention architecture. **No paper proposes a new attention mechanism designed for iterative denoising.** Our work is the natural architectural intervention following the 2601.07894 analysis — position as "they found the problem, we propose the fix."

**Why oral-level:** This is an ARCHITECTURE contribution — the rarest and most impactful type of paper. If mask-adaptive attention improves both quality AND speed (by skipping MASK-to-MASK attention), it becomes the default architecture for all future dLLMs.

**Method:** At each denoising step, construct a dynamic attention mask M where M[i,j] = 1 if token j is revealed OR token i is revealed, and M[i,j] = 0 if BOTH i and j are masked. This prevents masked tokens from attending to other masked tokens (which are all identical MASK embeddings — no information). Additionally, apply a learned gate g(t) that interpolates between sparse (early steps, high masking) and dense (late steps, low masking) attention based on the current denoising step t. Implement via a simple modification to the attention mask computation — no architectural changes to the Transformer blocks themselves.

**Implementation prompt:** Build a project that implements mask-adaptive attention for masked diffusion language models, replacing the standard full bidirectional attention with a dynamic attention mask that prevents masked tokens from attending to other masked tokens. This idea is verified novel — all existing dLLMs (LLaDA arXiv 2502.09992, Dream arXiv 2508.15487, MDLM arXiv 2406.07524, Mercury arXiv 2506.17298) use vanilla bidirectional Transformer attention regardless of masking state. Use LLaDA-8B-Instruct as the base model, modifying only the attention computation. The core modification: at each denoising step, compute a binary attention mask where entry M[i,j] = 0 if both position i and position j are currently masked (both hold the MASK token), and M[i,j] = 1 otherwise. This means revealed tokens attend to all other tokens (both revealed and masked), but masked tokens attend ONLY to revealed tokens (not to other masked tokens, which are identical uninformative MASK embeddings). Implement this by modifying the attention_mask passed to each Transformer layer in the HuggingFace LLaDA implementation — construct the mask dynamically at each denoising step based on which positions are currently unmasked. Additionally, implement a step-dependent attention temperature: at early denoising steps (high masking ratio), use sharper attention (temperature 0.5) to focus on the few revealed tokens; at late steps (low masking), use standard temperature (1.0). Train the modified LLaDA-8B on 10B tokens from SlimPajama with the mask-adaptive attention, comparing against the identical training run with standard attention. Evaluate on 6 benchmarks: GSM8K (1319 problems), MATH-500, AIME 2024 (30 problems), HumanEval (164 problems), GPQA Diamond (198 problems), and MT-Bench (80 prompts). Compare against 7 baselines: (1) standard LLaDA-8B with full bidirectional attention, (2) Dream-7B, (3) LLaDA with LogicDiff unmasking schedule, (4) LLaDA with linear attention approximation (Performer-style), (5) AR Llama-3.1-8B as autoregressive reference, (6) LLaDA with random attention dropout (drop 50% of attention connections randomly as a control), and (7) LLaDA with causal (left-to-right) attention mask. Run 8 ablations: (1) attention mask variants (our mask-adaptive versus full versus causal versus block-sparse), (2) step-dependent temperature (fixed 1.0 versus fixed 0.5 versus adaptive), (3) whether masked tokens should attend to OTHER masked tokens or not (the core design choice), (4) number of denoising steps (10, 20, 50, 100), (5) model scale (1.3B, 3B, 8B), (6) training data size (1B, 5B, 10B tokens), (7) gate interpolation (linear versus learned MLP versus step-function), and (8) combining mask-adaptive attention with LogicDiff unmasking. Analysis: measure FLOPs per denoising step (expect 30-50% reduction at early steps when 90% tokens are masked), plot attention entropy at each denoising step for standard versus mask-adaptive (mask-adaptive should have lower entropy = more focused), visualize attention maps at step 5 versus step 50 showing how attention patterns differ, and measure wall-clock inference speedup. The key results that validate the hypothesis are: (a) equal or better quality with fewer FLOPs, and (b) focused attention on informative tokens improves generation quality, especially for reasoning tasks.

## Idea 2: Long-Context Diffusion LMs via Step-Adaptive Sparse Attention

**Core insight:** dLLMs are limited to 4-8K context because full bidirectional attention at EVERY denoising step is quadratic × number-of-steps. No one has adapted sparse attention for dLLMs. Key insight: the optimal sparsity pattern CHANGES across denoising steps. At early steps (90% masked), attention should be very sparse (most tokens are MASK — no long-range dependencies needed). At late steps (10% masked), full attention is needed for coherence. This step-adaptive sparsity is unique to diffusion and doesn't exist in AR sparse attention.

**Novelty verification:** All sparse attention work (Native Sparse Attention, LongSight, sliding window) targets AR models. Zero papers adapt sparse attention specifically for masked diffusion LMs. LLaDA2.0 uses YaRN for 64K but this is RoPE extension, not sparse attention.

**Why oral-level:** Enables dLLMs at 32K-128K context for the first time. The step-adaptive insight is specific to diffusion — not a trivial transfer from AR sparse attention. Opens dLLMs to document-level tasks, long-form generation, and RAG.

**Implementation prompt:** Build a project that enables masked diffusion language models to process 32K-128K context via step-adaptive sparse attention, where the sparsity pattern changes across denoising steps. This idea is verified novel — all existing sparse attention methods (Native Sparse Attention, sliding window, LongSight) target autoregressive models, and no paper adapts sparse attention for the unique structure of iterative denoising in dLLMs. Use LLaDA-8B as the base, replacing standard full attention with a step-adaptive sparse attention mechanism. The design: define a step-dependent sparsity function S(t) that controls the attention radius at denoising step t. At step t with masking ratio r(t), use sliding window attention with window size W(t) = W_min + (W_max - W_min) * (1 - r(t))^2, where W_min=256 (early steps, high masking — only local attention needed because most tokens are uninformative MASK), W_max=full sequence length (final steps, low masking — full attention for global coherence). Additionally, add global attention tokens: at every step, the first and last 64 tokens always receive full attention (for instruction following and answer structure). Implement using FlashAttention-2's block-sparse attention API or xFormers' sparse attention. Train on SlimPajama with sequence lengths progressively increased: 8K for 5B tokens, then 32K for 3B tokens, then 64K for 1B tokens (following LLaDA2.0's YaRN approach but with sparse attention instead). Evaluate on 6 benchmarks: RULER (synthetic long-context, 4K to 128K), LongBench v2 (real-world long-context QA), InfiniteBench (100K+ context), MATH-500 (short-context baseline), HumanEval (code baseline), and a custom 32K-context code understanding task. Compare against 7 baselines: (1) standard LLaDA-8B (4K context), (2) LLaDA2.0 with YaRN 64K extension (if available), (3) AR Llama-3.1-8B-128K, (4) LLaDA with simple sliding window (fixed window, not step-adaptive), (5) LLaDA with random sparse attention (drop 50% of attention at all steps uniformly), (6) LLaDA with full attention at reduced batch size (memory-matched), and (7) Block Diffusion (BD3-LM) which handles arbitrary length via block-wise generation. Run 8 ablations: (1) window function W(t) shape (linear versus quadratic versus step-function), (2) W_min values (64, 128, 256, 512), (3) number of global attention tokens (0, 32, 64, 128), (4) context length (8K, 16K, 32K, 64K, 128K), (5) number of denoising steps at long context (10, 20, 50), (6) training curriculum (progressive versus direct 32K), (7) combining with RoPE NTK extension, and (8) step-adaptive versus fixed sparsity (ablating the core design). Analysis: plot perplexity versus context length for all methods, measure peak GPU memory versus context length showing the memory savings, plot the effective attention radius at each denoising step showing the adaptive pattern, benchmark wall-clock time versus context length, and run needle-in-a-haystack at 32K and 64K to test retrieval quality.

## Idea 3: Hybrid Causal-Diffusion Generation for Structured Agentic Output

**Core insight:** dLLMs fail catastrophically at agentic tasks (0% tool calling, <7.5% AgentBoard) because tool calls require CAUSAL structure (the function name must be generated before its arguments) and JSON requires SEQUENTIAL validity (brackets must match). Pure diffusion generates everything in parallel, violating these causal dependencies. Solution: a hybrid model that dynamically switches between causal (left-to-right) generation for structural tokens and diffusion (parallel) generation for content tokens. The model LEARNS which tokens need causal generation and which can be parallel.

**Novelty verification:** DiffuAgent (arXiv 2601.12979) documents the failure and proposes using dLLMs as plug-in modules but does NOT fix the agentic generation problem itself. S3 (arXiv 2507.04504) uses scaffolding for JSON but doesn't solve the fundamental causal-parallel tension. CARD (arXiv 2601.22031) proposes causal AR diffusion but for general text, not specifically for agentic structured output. No paper trains a dLLM to DYNAMICALLY switch between causal and diffusion modes based on content type.

**Why oral-level:** This addresses the #1 practical barrier to dLLM deployment. If dLLMs could handle tool calling and structured output, they'd immediately compete with AR models for agentic applications. The dynamic mode-switching mechanism is a genuine architectural contribution.

**Implementation prompt:** Build a project that enables diffusion language models to handle agentic tool use and structured output by dynamically switching between causal (left-to-right) generation for structural tokens and parallel diffusion generation for content tokens. This idea is verified novel — DiffuAgent (arXiv 2601.12979) documents that dLLMs achieve 0% tool calling success but proposes using them as plug-in modules rather than fixing the generation itself, and no paper trains a dLLM with dynamic causal-diffusion mode switching. Use LLaDA-8B-Instruct as the base. The core modification: add a binary mode predictor M(position) that outputs 0 (diffusion mode) or 1 (causal mode) for each token position. In causal mode, the token is generated left-to-right with causal attention mask. In diffusion mode, the token is generated via standard masked diffusion with bidirectional attention. The mode predictor is a lightweight 2-layer MLP on the first-layer hidden states, trained jointly with the main model. Training data construction: take 50K tool-calling examples from ToolBench and Gorilla, annotate each token as structural (function names, brackets, colons, commas, key names in JSON) or content (argument values, natural language). Structural tokens get mode=causal, content tokens get mode=diffusion. Train with a combined loss: causal cross-entropy for structural tokens + masked diffusion loss for content tokens. At inference: the mode predictor first classifies each position, then structural tokens are generated left-to-right while content tokens are filled in via diffusion in parallel. Evaluate on 6 benchmarks: (1) Berkeley Function Calling Leaderboard (BFCL, 2000+ test cases), (2) ToolBench (real API calls), (3) Gorilla API Bench (1645 API calls), (4) Spider SQL (1034 databases), (5) AgentBoard (9 tasks), and (6) MT-Bench (general quality, to check no degradation). Compare against 7 baselines: (1) standard LLaDA-8B-Instruct (pure diffusion), (2) LLaDA with S3 structured scaffolding, (3) LLaDA with DINGO grammar-constrained decoding (NeurIPS 2025), (4) LLaDA with Mundler et al. CFG-constrained decoding (ICLR 2026), (5) AR Llama-3.1-8B-Instruct (fully causal), (6) Block Diffusion BD3-LM (semi-autoregressive), and (7) LLaDA with post-hoc JSON fixing. Run 8 ablations: (1) mode predictor accuracy (oracle mode labels versus learned predictor versus random), (2) ratio of causal versus diffusion tokens (force 10%, 30%, 50%, 70% causal), (3) mode predictor architecture (MLP versus attention-based), (4) training data size (10K, 25K, 50K tool-call examples), (5) structural token definition (strict JSON syntax only versus including key names versus all non-value tokens), (6) with versus without pre-training the mode predictor, (7) model scale (3B versus 8B), and (8) combining with grammar-constrained decoding for the causal portion. Analysis: report JSON validity rate, function call accuracy, argument correctness separately, compare inference speed versus AR baseline (hybrid should be faster because content tokens are parallel), visualize the learned mode assignments showing what the model considers structural versus content, and measure the quality-structure tradeoff.

## Idea 4: Learned Token-Level Corruption via Differentiable Masking

**Core insight:** All masked diffusion LMs use UNIFORM random masking — every token has equal probability of being corrupted at each step. But not all tokens are equally important or equally difficult to predict. Function words ("the", "is") are trivial; content words ("Riemann", "hypothesis") carry all the information. Learning WHICH tokens to corrupt preferentially (and how much) should produce a tighter ELBO and better generation quality. Dream's CART adjusts the noise schedule per-token but uses a fixed heuristic (token frequency). We propose end-to-end learned token-level corruption via Gumbel-Softmax.

**Novelty verification:** BD3-LM (ICLR 2025) learns schedule parameters (min/max mask rate) but applies uniform masking within those bounds. Dream's CART reschedules based on token frequency (heuristic, not learned). The ICLR 2025 data-driven noise schedule paper optimizes schedule parameters to minimize training variance but does not learn per-token corruption probabilities. No paper learns a token-level corruption function end-to-end.

**Why oral-level:** This addresses the fundamental 16× compute gap between dLLMs and AR models. If the corruption process is optimized (not arbitrary), the ELBO bound tightens, requiring fewer denoising steps for the same quality. A tighter ELBO is a theoretical contribution; the practical speedup is the empirical contribution.

**Implementation prompt:** Build a project that learns the optimal per-token corruption probabilities for masked diffusion language models end-to-end, replacing uniform random masking with a learned corruption function. This idea is verified novel — BD3-LM (ICLR 2025) learns schedule parameters but applies uniform masking within bounds, Dream's CART uses token-frequency heuristics, and no paper learns a token-level corruption function end-to-end via gradient descent. Use LLaDA-8B architecture as the base. The core modification: add a corruption network C(x, t) that takes the clean sequence x and timestep t, and outputs per-token masking probabilities p_i in [0,1] for each position i. C is a lightweight 2-layer Transformer (4 layers, 256 hidden dim, ~10M params) that processes the clean token embeddings and outputs masking logits. During training, sample the masking decisions via Gumbel-Softmax (temperature 0.5) to maintain differentiability. The corruption network is trained jointly with the denoising model: the total loss is the standard MDLM denoising loss, but the corruption pattern is produced by C rather than sampled uniformly. The gradient flows through the Gumbel-Softmax samples back to C, optimizing it to produce corruption patterns that minimize the denoising loss (i.e., patterns that make the denoising model's job easier = tighter ELBO). Key regularization: add an entropy penalty to prevent C from always masking the same tokens, ensuring diverse corruption patterns. Also add a constraint that the AVERAGE masking rate matches the schedule (C adjusts which tokens to mask, not how many). Train on SlimPajama 10B tokens. Evaluate on 6 benchmarks: GSM8K, MATH-500, HumanEval, GPQA Diamond, MT-Bench, and generative perplexity on held-out C4. Compare against 7 baselines: (1) standard uniform masking (LLaDA default), (2) Dream's CART token-frequency-based schedule, (3) BD3-LM's learned clipped schedule, (4) inverse-frequency masking (mask rare tokens more — a simple heuristic), (5) entropy-based masking (mask high-entropy tokens more), (6) random non-uniform masking (sample p_i from Beta distribution), and (7) AR Llama-3.1-8B (compute-matched). Run 8 ablations: (1) corruption network size (2-layer MLP versus 2-layer Transformer versus 4-layer Transformer), (2) Gumbel-Softmax temperature (0.1, 0.5, 1.0, 2.0), (3) entropy regularization weight (0.01, 0.1, 1.0), (4) whether C sees the clean tokens or only their embeddings, (5) fixed average mask rate versus learned average rate, (6) number of denoising steps at inference (5, 10, 20, 50 — does learned corruption help more at fewer steps?), (7) training data size (1B, 5B, 10B tokens), and (8) model scale (1.3B, 3B, 8B). Analysis: visualize the learned corruption patterns — which tokens does C prefer to mask? (expect: content words masked more at early steps, function words at late steps), measure ELBO gap between learned corruption and uniform corruption (the tighter the ELBO, the better), plot quality-versus-steps curve showing learned corruption achieves same quality with fewer steps, and analyze whether the corruption network discovers linguistically meaningful patterns (POS-tag correlation with masking probability).

## Idea 5: Diffusion LMs for Code Diff Generation

**Core insight:** Current code agents generate FULL files, even when only a few scattered lines need to change. This is wasteful and error-prone — the model must reproduce thousands of unchanged lines perfectly. Diffusion LMs have a unique advantage: they can generate text at ARBITRARY positions in parallel, making them ideal for generating scattered DIFFS (minimal edits across a file) rather than full files. The model takes (original code + edit instruction) and outputs ONLY the changed lines with their positions.

**Novelty verification:** DiffuCoder (Apple, arXiv 2506.20639), CoDA (Salesforce), and Dream-Coder all do full code GENERATION with diffusion, not diff/patch generation. GPT-5.1's apply_patch tool is AR-based. JetBrains' blog mentions diffusion for code editing as a future direction but no paper implements it. No paper uses dLLMs specifically for generating minimal code diffs.

**Why oral-level:** Code editing is the #1 use case for coding assistants. Generating diffs instead of full files would be faster (fewer tokens), more accurate (fewer chances to corrupt unchanged code), and more interpretable (users see exactly what changed). The connection to diffusion's parallel nature is clean and elegant.

**Implementation prompt:** Build a project that uses masked diffusion language models to generate code DIFFS (minimal edits at scattered positions across a file) rather than full code, leveraging diffusion's ability to generate tokens at arbitrary positions in parallel. This idea is verified novel — DiffuCoder (Apple, arXiv 2506.20639), CoDA (Salesforce, arXiv 2510.03270), and Dream-Coder (arXiv 2509.01142) all do full code generation with diffusion, and no paper uses dLLMs specifically for diff/patch generation. Use LLaDA-8B as the base model. The method: given original code C and an edit instruction I, the model's input is the concatenation [I; C] where C is the full original code. The MASK tokens are placed ONLY at positions where edits are needed — the unchanged code remains visible as context. The model then fills in the masked positions with the new code via standard denoising. The key challenge: the model must PREDICT which positions need editing (not given a priori). Solution: two-stage inference — Stage 1 (Edit Localization): the model predicts a per-token edit probability using a binary classification head on the hidden states, identifying which lines/tokens need to change. Stage 2 (Diff Generation): mask the identified positions and denoise to generate the new code. The edit localization head is trained on code diff data where the changed positions are known from git diffs. Training data: extract 500K (original_file, edit_instruction, diff) triples from GitHub commits. For each, the instruction is the commit message, the diff marks which lines changed. Train with two losses: edit localization loss (binary CE per token) + masked diffusion loss (standard MDLM loss on the masked-then-denoised edit positions). Evaluate on 6 benchmarks: (1) SWE-Bench Verified (300 issues — generate diffs instead of full files), (2) SWE-Bench Lite (25 issues), (3) Aider Polyglot benchmark (133 tasks across Python/JS/Java), (4) EditEval (code editing evaluation from CMU), (5) CanItEdit (code editing benchmark), and (6) HumanEval (full generation baseline to check no degradation). Compare against 7 baselines: (1) AR Llama-3.1-8B generating full files, (2) AR Llama-3.1-8B with search-and-replace prompt (generate diffs as text), (3) GPT-5.1 apply_patch tool, (4) LLaDA-8B generating full files (standard diffusion baseline), (5) DiffuCoder (Apple, full code generation), (6) Aider's diff generation (AR-based), and (7) our model WITHOUT the edit localization stage (mask random positions instead of predicted edit positions). Run 8 ablations: (1) edit localization threshold (what probability counts as "needs editing"), (2) context window for unchanged code (full file versus ±50 lines around edits), (3) training data size (50K, 100K, 250K, 500K commits), (4) diff granularity (line-level versus token-level masking), (5) number of denoising steps for diff generation (5, 10, 20), (6) with versus without edit localization stage (oracle edit positions versus predicted versus random), (7) multi-file diffs (changes across 2-3 files simultaneously), and (8) programming language (Python versus JavaScript versus Java). Analysis: measure edit precision (% of predicted edit positions that actually needed editing), edit recall (% of actual edit positions correctly identified), tokens generated per task (diffs should be 5-20x fewer than full files), wall-clock time comparison, and qualitative examples showing scattered edits across a file generated in parallel versus AR sequential generation. The key result: equal or better edit accuracy with 5-20x fewer generated tokens and proportional speedup.
